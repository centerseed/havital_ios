## 詳細驗證

### BUG-01：登出後未導向 LoginView

**驗證步驟**：
1. 主頁 → ⋯ → 個人資料 → 滑到底部 → 點「登出」
2. 觀察畫面即時反應
3. 手動重啟 App，觀察是否出現 LoginView

**驗證結果**：⚠️ 部分修復

| 行為 | 結果 |
|------|------|
| 登出後 auth state 清除 | ✅ 有效（重啟 App 後顯示 LoginView） |
| 登出後即時導向 LoginView | ❌ 仍未發生（App 停留在訓練主頁，profile sheet 自動關閉） |

**實際行為**：點擊登出後，個人資料 sheet 關閉，回到訓練主頁。需手動重啟才能看到 LoginView。
**預期行為**：點擊登出後，立即顯示 LoginView，無需重啟。

**結論**：狀態清除部分已修復，但 UI 導向邏輯仍有問題。

### BUG-02：Token 過期無自動更新

**驗證結果**：⏭️ 無法在 Demo Mode 測試

Demo Mode 帳號無真實 auth token，無法重現 token 過期情境。需使用一般帳號（email/password 登入）並等待 token 過期才可驗證。

### BUG-05：產生下週課表後跳至錯誤週數

**驗證步驟**：
1. 主頁確認在第 1 週（本週）
2. ⋯ → 切換週數 → 選第 2 週
3. 第 2 週顯示「週課表尚未產生」
4. 點擊「產生週課表」按鈕
5. 等待生成完成，觀察最終停在哪週

**驗證結果**：❌ 未修復（且更嚴重）

| 行為 | 原始 Bug | 本次結果 | 預期 |
|------|---------|---------|------|
| 產生第 2 週後停在哪週 | 第 3 週 | **第 4 週** | 第 2 週 |

**實際行為**：產生第 2 週課表後，view 跳至第 4 週（03/16-03/22，巔峰期 第3-4週）。
**預期行為**：產生後應停留在剛產生的第 2 週。

本次跳轉行為比原始 Bug 更嚴重（跳 3 週而非 1 週）。推測可能與已有週數的快取邏輯有關。

## 給工程團隊的建議

### 🔴 仍需修復

**BUG-01（部分）** — auth state 清除有效，但 `authViewModel.isAuthenticated` 的 `@Published` 變更未即時觸發 ContentView 重新渲染切換到 LoginView。建議檢查 logout handler 是否在 `@MainActor` 上執行，並確保 `isAuthenticated = false` 後 SwiftUI state 能即時更新。

**BUG-04** — 兩頁面使用不同的數值計算邏輯（可能一個用 `Int()` 截斷，另一個用 `round()`）。建議統一為同一個計算函式，並新增單元測試驗證 5 個區間邊界連續無空隙。

**BUG-05** — 生成後的導航邏輯問題加重。建議在 `GenerateNextWeekButtonV2` 的完成 callback 中，明確設定 `viewModel.selectedWeek = generatedWeekNumber`（API 回傳的週數），而非依賴 API 預設的導航行為。

### ⏭️ 待確認

**BUG-02** — 需使用真實帳號（email 登入）在 token 過期後測試。建議提供測試帳號給 QA 進行完整驗證。
