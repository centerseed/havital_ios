# Feature Spec: 編輯 V2 課表——力量訓練支援

## 背景與動機

目前 V2 課表編輯功能在加入力量訓練後出現兩個根本性問題：

**問題一：力量訓練根本無法編輯**
編輯畫面沒有實作力量訓練的欄位（strength type、動作清單、組數/次數/重量），使用者進入力量日的編輯介面看到的是跑步欄位（距離、配速），無法輸入任何有意義的力量訓練內容。

**問題二：拖拉調換天次後主訓練消失**
當某天有跑步（主訓練）＋力量（補充訓練）時，與另一天拖拉調換後，跑步主訓練消失，只保留部分資料，導致課表資料遺失。

這兩個問題讓力量訓練功能對使用者而言幾乎不可用。

## 目標用戶

每週安排跑步課表並搭配 1–3 天力量訓練的跑者，需要能自行調整課表內容（動作、組次、天次排列）的自訓型使用者。

## 需求

### P0（必須有）

#### R1. 力量訓練日可正確編輯

- **描述**：當日類型為力量訓練（category = strength）時，編輯介面顯示力量訓練欄位，而非跑步欄位。使用者可以編輯：訓練重點（strength_type）、動作清單（名稱、組數、次數/時間、重量）。
- **Acceptance Criteria**：
  - Given 當日類型為力量訓練，When 使用者進入編輯，Then 主要輸入欄位為力量訓練欄位（strength type 選擇、動作清單），不顯示距離/配速欄位
  - Given 使用者修改 strength_type，When 儲存，Then 週課表反映新的訓練重點
  - Given 使用者新增一個動作（設定名稱 + 組數 + 次數），When 儲存，Then 動作出現在當日詳細頁

#### R2. 任何天次可新增力量訓練

- **描述**：使用者可在訓練日（跑步/交叉訓練）或休息日新增力量訓練。訓練日新增後，力量訓練以補充訓練（supplementary）形式附加；休息日新增後，力量訓練成為該日主訓練。
- **Acceptance Criteria**：
  - Given 當日類型為跑步，When 使用者新增力量訓練，Then 力量訓練以補充訓練身份出現在該日
  - Given 當日類型為休息，When 使用者新增力量訓練，Then 該日轉為力量訓練日，力量成為主訓練
  - Given 使用者新增力量訓練但未輸入任何動作，When 嘗試儲存，Then 系統提示至少需要一個動作

#### R3. 任何天次可刪除力量訓練

- **描述**：使用者可從訓練日移除補充力量訓練，或將力量訓練日改回休息日。
- **Acceptance Criteria**：
  - Given 當日有補充力量訓練，When 使用者刪除，Then 補充訓練從該日移除，主訓練維持不變
  - Given 當日類型為力量訓練，When 使用者刪除力量訓練，Then 該日轉為休息日

#### R4. 動作層級可編輯

- **描述**：使用者可新增、刪除、編輯力量訓練中的每一個動作，每個動作可設定：名稱（必填）、組數、次數（如 "8-12"）或持續時間（秒）、重量（kg）。
- **Acceptance Criteria**：
  - Given 使用者編輯某個動作的組數，When 儲存，Then 詳細頁顯示更新後的組數
  - Given 使用者刪除一個動作，When 儲存，Then 該動作不再出現在詳細頁
  - Given 使用者新增一個動作並只填名稱，When 儲存，Then 動作正確新增（其他欄位為空）

#### R5. 拖拉調換天次完整保留資料

- **描述**：拖拉兩天互換位置時，每一天的完整資料（主訓練 + 補充訓練）必須完整對調，不得遺失任何一方的資料。
- **Acceptance Criteria**：
  - Given A 天有跑步（主）＋力量（補充），B 天為休息，When 拖拉 A、B 互換，Then A 天變休息，B 天有跑步（主）＋力量（補充），兩邊資料均完整
  - Given A 天有跑步（主）＋力量（補充），B 天有另一個跑步（主），When 拖拉互換，Then A 取得 B 的跑步主訓練，B 取得 A 的跑步主訓練＋力量補充，無任何遺失
  - Given 任意兩天互換，When 完成後立即查看週課表，Then 兩天的天目標（day_target）、原因（reason）、訓練內容均已正確對調

### P1（應該有）

#### R6. 動作教學說明可查閱

- **描述**：在力量訓練編輯或詳細頁中，若動作有對應的教學圖示，使用者點擊動作名稱可彈出教學說明（保留現有 ExerciseInstructionView 行為）。
- **Acceptance Criteria**：
  - Given 動作有對應的 exerciseId 映射，When 使用者點擊動作名稱，Then 彈出教學說明 sheet

#### R7. 未儲存變更保護

- **描述**：使用者在有未儲存變更的情況下嘗試離開編輯頁面時，系統提示確認是否放棄變更。
- **Acceptance Criteria**：
  - Given 使用者有未儲存變更，When 嘗試返回，Then 系統顯示「放棄變更？」提示，選取放棄才離開

## 明確不包含

- 不實作訓練完成紀錄（打勾、組次確認、實際重量回填）
- 不改變週課表列表（週視圖）的顯示方式
- 不改變力量訓練詳細頁（非編輯模式）的呈現
- 不定義後端課表生成邏輯（由 SPEC-weekly-plan-v2 定義）
- 不實作引導式訓練（step-by-step 動作引導）

## 技術約束（給 Architect 參考）

- 拖拉互換邏輯需完整交換 `session`（含 `supplementary`）與 `day_target`、`reason`、`tips`、`category`，不得只交換部分欄位
- 力量訓練編輯後提交的 API payload 必須符合 `StrengthActivityDTO` 結構（strength_type + exercises[]）
- 編輯畫面需支援在同一個 UI 框架中切換「跑步模板」與「力量模板」，需確保跑步編輯既有能力不退化
- 休息日新增力量訓練時，需更新 `category` 為 `strength`

## 開放問題

1. 使用者在力量日新增補充跑步（如短暫熱身慢跑）是否需要支援？目前暫不定義。
2. 力量訓練動作名稱是否需要從預設清單選取，還是允許自由輸入？目前假設自由輸入。
3. strength_type 是否允許使用者自定義，還是只能從固定清單（5 種）選取？

## Priority

High — 力量訓練是課表核心功能之一，目前等同不可用。
