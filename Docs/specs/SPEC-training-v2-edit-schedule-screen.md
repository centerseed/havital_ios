# SPEC: 編輯 v2 課表畫面（含肌力訓練編輯）

## Background and motivation

目前 v2 編輯畫面在「肌力訓練」情境存在明顯語意錯配：

- 畫面標題為「肌力訓練」，主要輸入卻是「距離（公里）」
- 使用者無法在主流程中直接表達肌力訓練的核心內容（部位、組數、次數、負重、器材）
- 同一套編輯介面同時承擔跑步訓練與肌力訓練，導致欄位意義不一致

這會造成三個產品風險：

1. 使用者不信任課表（內容看起來「不專業」或「不合理」）
2. 編輯完成後難以執行（不知道實際要做哪些動作）
3. 教練語意與使用者語意不一致（系統說肌力、實際像跑步）

本 Spec 的目的，是定義「編輯 v2 課表」在肌力訓練場景下的正確體驗，讓畫面語意與訓練語意一致。

## Target users

1. 已有跑步課表，且每週需安排 1-3 天肌力訓練的跑者
2. 需要根據身體狀態快速調整當週課表的跑者
3. 希望「看得懂、能執行」訓練內容的自訓型使用者

## Requirements

### R1. 訓練類型語意一致

- 當日類型為「肌力訓練」時，畫面不得以「距離（公里）」作為唯一或主要輸入
- 肌力訓練需使用肌力語意欄位（例如：訓練重點、動作清單、組次資訊）
- 跑步訓練維持跑步語意欄位（例如：距離、配速、暖身、緩和）

### R2. 依訓練類型切換編輯模板

- 「跑步型」與「肌力型」需使用不同編輯模板
- 使用者切換訓練類型時，畫面需即時切換對應欄位
- 切換後若有不相容資料，需清楚提示處理方式（保留、轉換或移除）

### R3. 肌力訓練最小可執行內容

- 肌力編輯完成後，使用者可在日卡直接理解「今天要做什麼」
- 每個肌力訓練日至少可表達：
1. 訓練重點（例如：下肢力量、核心穩定）
2. 動作集合（至少 1 個）
3. 每個動作的工作量（以組/次/時間其中一種可執行格式呈現）

### R4. 週視圖可讀性

- 在週課表列表中，肌力日與跑步日須可一眼區分
- 肌力日摘要應顯示肌力語意，不得退化成跑步語意（例如僅顯示 km）
- 使用者不進入詳細編輯也能理解該日大致訓練內容

### R5. 變更安全與可回退

- 編輯流程需維持取消、放棄變更、儲存三種明確狀態
- 使用者離開前若有未儲存變更，必須有提示
- 儲存成功後，週課表立即反映新摘要內容

### R6. 文案與認知負擔

- 欄位名稱需符合一般跑者語言，不使用不必要術語
- 同一欄位名稱在不同訓練類型中不得有衝突意義
- 使用者在 10 秒內可辨識「這是肌力編輯，不是跑步編輯」

## Acceptance criteria

### AC1. 肌力日欄位合理性

- Given 當日類型為肌力訓練
- When 使用者進入編輯
- Then 主編輯欄位不以「距離（公里）」為核心，且可輸入肌力訓練工作量

### AC2. 類型切換一致性

- Given 使用者在同一天於跑步型與肌力型之間切換
- When 完成切換
- Then 編輯欄位會對應切換，且系統明確處理不相容資料

### AC3. 週卡摘要可執行

- Given 使用者儲存肌力訓練日
- When 回到週課表
- Then 該日摘要可直接指導執行（可看出重點與工作量），不只顯示距離或空白描述

### AC4. 未儲存保護

- Given 使用者有未儲存變更
- When 嘗試離開畫面
- Then 系統提示放棄或繼續編輯，不會靜默遺失

### AC5. 快速辨識

- Given 新使用者首次看到肌力日編輯
- When 閱讀畫面 10 秒
- Then 能正確說出這是肌力訓練編輯而非跑步課表編輯

## PLAN scope

Plan ID: `plan-training-v2-edit-screen-rationalization`

- P1 問題定義與體驗對齊：確認 v1/v2 目前編輯語意衝突與優先修復範圍
- P2 肌力訓練編輯體驗：定義肌力模板、摘要規則、欄位語意
- P3 跑步與肌力共存規則：定義類型切換、資料轉換、錯誤防呆
- P4 驗證與上線門檻：建立 QA 驗收腳本與回歸清單

Plan exit criteria:

1. 上述 AC1-AC5 全部可在 QA 流程中逐條驗證
2. 肌力日不再以距離公里作為主要輸入
3. 週視圖中肌力日摘要可執行且可辨識

## Explicitly out of scope

- 不定義 API 欄位命名與資料 schema
- 不定義資料庫 migration 策略
- 不定義前端框架或元件實作細節
- 不處理非課表編輯場景（例如訓練記錄上傳、分享卡）

## Technical constraints for Architect

- 必須保留既有「未儲存變更保護」的體驗完整性
- 必須支援週課表層級的即時摘要刷新
- 必須確保跑步型與肌力型在同一週編輯流程可共存
- 需提供回歸測試邊界，避免破壞既有跑步編輯能力

## Open questions

1. 肌力訓練是否需要區分「自重/器械/槓鈴」子類型，作為預設模板入口？
2. 肌力訓練日是否允許同時包含短跑步內容（例如熱身慢跑），若允許要如何在摘要呈現？
3. 週卡摘要是否需要顯示完成狀態（已完成組次）或僅顯示計畫內容？
4. v1 既有使用者在升級到 v2 時，舊有肌力日資料如何對齊新語意？

## Priority

- Priority: High
- 期望影響：降低課表編輯困惑、提升肌力日可執行性、改善專業可信度
